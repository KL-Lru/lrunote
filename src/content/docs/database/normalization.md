## なぜ正規化するのか

> One Fact in One Place

正規化はデータベース設計において, 1 つの事実は 1 つの場所にのみ存在するように制御することにより, データが矛盾することを回避し, 効率的に管理できるようにすることを目的とする.

## 前提知識

### キーと呼ばれる属性たち

データベーステーブル中で, 一意にレコードを識別するための最少数の属性の組み合わせを「候補キー(Candidate Key)」と呼ぶ.
そのうち 1 つの主たるものを選択したものが「主キー(Primary Key)」となる.
主キーとして選択されなかった候補キーは「代替キー(Alternate Key)」と呼ぶ.
いかなる候補キーにもならない属性は「非キー属性(Non-key Attribute)」と呼ぶ.

データ上に自然に存在しているもので候補キーを構成できる場合, それを「自然キー(Natural Key)」と呼ぶ.
自然に候補キーが存在しない場合や, 意図的に自然キーを避ける場合, 連番や UUID などの人為的な候補キーを作成できる.
このように作成された候補キーは「代用キー(Surrogate Key)」と呼ぶ.

ある属性が他のテーブルの属性の値を参照し, その間に関係を持つ場合, 参照される側の属性を「外部キー(Foreign Key)」と呼ぶ.

### 関数従属

あるテーブルにおいて, 1 つの属性 A が決まると他の属性 B が一意に定まるような関係を「関数従属(Functional Dependency)」と呼ぶ.
主キー以外の属性からの関数従属を持つ属性が存在する場合, そこには推移的な関数従属(Transitive Dependency)が存在する.

属性 B が属性 A に関数従属する場合, 次のように記述される.

```code
A -> B
```

関数従属性には次のような性質がある.

| 性質   | 内容                                                      |
| ------ | --------------------------------------------------------- |
| 反射律 | `Y ⊆ X ⊆ U` の場合, `X -> Y` は自明に成り立つ             |
| 増加律 | `X -> Y` かつ `Z ⊆ U` の場合, `X ∪ Z -> Y ∪ Z` も成り立つ |
| 推移性 | `X -> Y`, `Y -> Z` の場合, `X -> Z` も成り立つ            |
| 合併律 | `X -> Y`, `X -> Z` の場合, `X -> Y ∪ Z` も成り立つ        |
| 分解律 | `X -> Y ∪ Z` の場合, `X -> Y` かつ `X -> Z` も成り立つ    |

非キー属性がキーの真部分集合に関数従属することを「部分関数従属(Partial functional dependency)」と呼ぶ.
逆に, キーの真部分集合に関数従属しない場合, それを「完全関数従属(Full functional dependency)」と呼ぶ.

## 正規系

### 第一正規形

関係 R が繰り返し項目を 1 つも含まないものを第一正規形という.

### 第二正規形

関係 R が第一正規形であり, かつ非キー属性が候補キーに完全関数従属する場合, それを第二正規形という.

### 第三正規形

関係 R が第二正規形であり, かつ非キー属性が候補キーに推移的に関数従属しない場合, それを第三正規形という.

### ボイス・コッド正規形

関係 R のすべての候補キーについて, その候補キー以外の属性が候補キーに完全関数従属する場合, それをボイス・コッド正規形という.

### 第四正規形

関係 R の中に多値従属性 A ->-> B が存在し, 常に R のすべての属性 X が A に関数従属する場合, それを第四正規形という.
候補キー A があり, 多値従属の場合はその集合が, 関数従属の場合はその値が, それぞれ 1 つに定まる場合である.

### 第五正規形

関係 R が結合従属性を持たず, 再結合することで元の関係を復元できる形態に分解した形式を第五正規形という.
